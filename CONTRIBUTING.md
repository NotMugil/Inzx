\## Ways to Contribute



\### üêõ Opening an Issue



Issues are the primary way to report bugs, request new features, or ask questions.



Before opening a new issue:

\- Search existing issues to avoid duplicates.

\- Use the provided \[\*\*ISSUE\_TEMPLATE\*\*](https://github.com/nirmaleeswar30/Inzx/tree/main/.github/ISSUE\_TEMPLATE).

\- Include clear and relevant details, such as:

&nbsp; - Steps to reproduce (for bugs)

&nbsp; - Expected vs actual behavior

&nbsp; - Screenshots, logs, or screen recordings if applicable

&nbsp; - Environment details (device, OS, Flutter/Dart version)



Clear and well-structured issues help maintainers respond more effectively.



\### üîÄ Contributing via Fork \& Pull Request



Code and documentation contributions should follow the fork-based workflow:



1\. Fork the repository to your GitHub account.

2\. Create a new branch for your changes:

&nbsp;  ```bash

&nbsp;  git checkout -b feature/your-feature-name

&nbsp;  ```

3\. Make your changes, keeping them focused and well-structured.

4\. Commit with a clear and meaningful message:

&nbsp;  ```

&nbsp;  git commit -m "Add: short description of change"

&nbsp;  ```

5\. Push the branch to your fork:

&nbsp;  ```

&nbsp;  git push origin feature/your-feature-name

&nbsp;  ```

6\. Open a Pull Request and describe your changes clearly.

7\. Reference related issues if applicable.

8\. Include screenshots or recordings for UI changes.



---



\## Development Guidelines



To maintain code quality and consistency:



\- Follow the \[Dart Style Guide](https://dart.dev/guides/language/effective-dart/style)

\- Write concise, descriptive commit messages

\- Add or update tests when introducing new features

\- Update documentation if behavior or APIs change

\- Run the following before submitting:

&nbsp; ```

&nbsp; dart format .

&nbsp; flutter analyze

&nbsp; ```

\- Ensure all checks pass before opening a Pull Request
